Conceptos clave (qué es cada cosa)

Mesa y Sesión de mesa (los dos modelos)

▼

Mesa (mesas): la mesa física del bar. Campos: numero (único, entero), pin (4 dígitos), estado (disponible / ocupada), activa (si está habilitada en el salón).

backend/src/models/Mesa.js 
const Mesa = sequelize.define('Mesa', {
  id:     { type: DataTypes.INTEGER, primaryKey: true, autoIncrement: true },
  numero: { type: DataTypes.INTEGER, allowNull: false, unique: true },
  pin:    { type: DataTypes.STRING,  allowNull: false },
  estado: { type: DataTypes.STRING,  defaultValue: 'disponible' },
  activa: { type: DataTypes.BOOLEAN, defaultValue: true }
}, { tableName: 'mesas', timestamps: false });

Sesión de mesa (sesiones_mesa): representa una atención, desde que se abre la mesa hasta que se cierra. Una mesa tiene muchas sesiones a lo largo del tiempo. Los pedidos se cuelgan de la sesión (sesionMesaId), no de la mesa directamente.

backend/src/models/SesionMesa.js 
const SesionMesa = sequelize.define('SesionMesa', {
  id:            { type: DataTypes.INTEGER, primaryKey: true, autoIncrement: true },
  fechaApertura: { type: DataTypes.DATE, allowNull: false, defaultValue: DataTypes.NOW },
  fechaCierre:   { type: DataTypes.DATE },  // null mientras la sesion sigue abierta
  estado:        { type: DataTypes.STRING, defaultValue: 'abierta' }  // abierta / cerrada
}, { tableName: 'sesiones_mesa', timestamps: false });
backend/src/models/index.js (relaciones) 
// una mesa tiene muchas sesiones, y cada sesion pertenece a una mesa
Mesa.hasMany(SesionMesa, { foreignKey: "mesaId" });
SesionMesa.belongsTo(Mesa, { foreignKey: "mesaId" });

// los pedidos se cuelgan de la sesion (no de la mesa directamente)
SesionMesa.hasMany(Pedido, { foreignKey: "sesionMesaId" });
Pedido.belongsTo(SesionMesa, { foreignKey: "sesionMesaId" });

// la mesa tambien tiene muchas cuentas (cada cliente arma la suya)
Mesa.hasMany(Cuenta, { foreignKey: "mesaId" });
Cuenta.belongsTo(Mesa, { foreignKey: "mesaId" });
Por qué dos modelos: una mesa se usa muchas veces. Cada atención (de apertura a cierre) es una sesión distinta. Así los pedidos quedan asociados a la sesión y se puede tener historial por atención.

PIN, QR y Token de mesa

▼

PIN

4 dígitos al azar. Es lo que el cliente necesita además del QR para entrar. Se genera nuevo cada vez que se abre la mesa, y al cerrar se pone en '0000' para bloquear el acceso.

QR

Apunta a la URL del cliente (origin/numero). No se guarda en la BD: se arma al vuelo con un servicio externo (api.qrserver.com) pasándole esa URL.

Token de mesa (JWT temporal, 8h)

Distinto del token de login del personal. Lleva tipo: "mesa", mesaId, numero, sesionMesaId. Autoriza al cliente en las rutas protegidas (cuentas, carrito, pedidos), sin que el cliente sea un usuario con contraseña.

backend/src/routes/table.routes.js 
// pin random de 4 digitos
const generarPinMesa = () => Math.floor(1000 + Math.random() * 9000).toString();

// token temporal de la mesa con el que despues entra el cliente
const generarMesaToken = (mesa, sesionMesa) => {
  return jwt.sign(
    { tipo: "mesa", mesaId: mesa.id, numero: mesa.numero, sesionMesaId: sesionMesa.id },
    JWT_SECRET,
    { expiresIn: MESA_TOKEN_EXPIRES }  // "8h"
  );
};
3

Ciclo de vida de una mesa

Paso 1 — Crear la mesa (admin)

▼

En MesasAdmin, "Agregar mesa" → guardarMesa → POST /mesas. Se crea con estado: 'disponible', activa: true y un PIN automático. Antes de crear se chequea que el número no exista.

backend/src/routes/table.routes.js — POST /mesas 
router.post("/mesas", verificarToken, verificarRol(['super-admin', 'admin']), async (req, res) => {
  try {
    const validacion = validarMesa(req.body);
    if (validacion.error) return res.status(400).json({ mensaje: validacion.mensaje });

    const { numero } = req.body;

    // si ya existe una mesa con ese numero avisamos claro (no un 500 generico)
    const mesaExistente = await Mesa.findOne({ where: { numero } });
    if (mesaExistente) {
      return res.status(400).json({ mensaje: `Ya existe una mesa con el número ${numero}` });
    }

    const pinAleatorio = generarPinMesa();
    const mesa = await Mesa.create({ numero, pin: pinAleatorio });
    res.status(201).json(mesa);
  } catch (error) {
    res.status(500).json({ mensaje: "Error al crear la mesa" });
  }
});

Paso 2 — Abrir la mesa (cuando llegan clientes)

▼

Botón "Abrir mesa" → POST /mesas/:id/abrir. Hace 3 cosas: pone estado: 'ocupada', genera un PIN nuevo y crea una SesionMesa (estado: 'abierta', fechaApertura, usuarioAperturaId = quién la abrió).

backend/src/routes/table.routes.js — POST /mesas/:id/abrir 
router.post("/mesas/:id/abrir", verificarToken, async (req, res) => {
  const mesa = await Mesa.findByPk(req.params.id);
  if (!mesa) return res.status(404).json({ mensaje: "Mesa no encontrada" });
  if (mesa.estado === 'ocupada') return res.status(400).json({ mensaje: "La mesa ya está abierta" });

  const nuevoPin = generarPinMesa();
  await mesa.update({ estado: 'ocupada', pin: nuevoPin });

  // arranca una sesion para esta atencion
  await SesionMesa.create({
    mesaId: mesa.id,
    usuarioAperturaId: req.usuario.id,
    estado: 'abierta',
    fechaApertura: new Date()
  });

  res.json({ mensaje: "Mesa abierta", mesa });
});

Paso 3 — El cliente entra (valida el PIN)

▼

Escanea el QR → va a /:numero → ValidarMesa → escribe el PIN → POST /mesas/numero/:numero/validar. El backend chequea: que la mesa esté ocupada, que el PIN coincida y que haya una sesión abierta. Si todo está bien, devuelve el token de mesa.

backend/src/routes/table.routes.js — POST /mesas/numero/:numero/validar 
router.post("/mesas/numero/:numero/validar", async (req, res) => {
  const { numero } = req.params;
  const { pin } = req.body;

  const mesa = await Mesa.findOne({ where: { numero } });
  if (!mesa) return res.status(404).json({ mensaje: "Mesa no encontrada" });

  if (mesa.estado !== 'ocupada') return res.status(403).json({ mensaje: "La mesa no está abierta..." });
  if (mesa.pin !== pin)          return res.status(401).json({ mensaje: "PIN incorrecto" });

  const sesionMesa = await SesionMesa.findOne({ where: { mesaId: mesa.id, estado: "abierta" } });
  if (!sesionMesa) return res.status(403).json({ mensaje: "La mesa no tiene una sesion abierta" });

  const mesaToken = generarMesaToken(mesa, sesionMesa);
  res.json({ mensaje: "Acceso concedido", mesaId: mesa.id, numero: mesa.numero, mesaToken });
});
El front guarda ese token con iniciarSesionMesa del Context (que persiste en sessionStorage). Desde ahí el cliente ve el menú, crea cuentas, arma el carrito y confirma pedidos (todo ligado a sesionMesaId).

Paso 4 — Cerrar la mesa (admin)

▼

Botón "Cerrar mesa" (con modal de confirmación) → POST /mesas/:id/cerrar. Pone la mesa en disponible, el PIN en '0000' (corta el acceso), pasa la sesión a cerrada (con fechaCierre) y cierra todas las cuentas abiertas de esa mesa.

backend/src/routes/table.routes.js — POST /mesas/:id/cerrar 
router.post("/mesas/:id/cerrar", verificarToken, async (req, res) => {
  const mesa = await Mesa.findByPk(req.params.id);
  if (!mesa) return res.status(404).json({ mensaje: "Mesa no encontrada" });

  // libera la mesa y bloquea el acceso invalidando el pin
  await mesa.update({ estado: 'disponible', pin: '0000' });

  // cierra la sesion que estaba abierta
  const sesionAbierta = await SesionMesa.findOne({ where: { mesaId: mesa.id, estado: 'abierta' } });
  if (sesionAbierta) await sesionAbierta.update({ estado: 'cerrada', fechaCierre: new Date() });

  // cierra todas las cuentas que quedaron abiertas
  await Cuenta.update({ estado: 'cerrada' }, { where: { mesaId: mesa.id, estado: 'abierta' } });

  res.json({ mensaje: "Mesa cerrada correctamente", mesa });
});

Panel del admin (MesasAdmin)

La tabla, el formulario y los handlers

▼

La tabla tiene una fila por mesa con columnas: Mesa (#número), Estado (badge), PIN cliente (+ botón "Nuevo PIN"), QR (botón "Abrir QR" + "Ir a mesa"), Sesión (botón "Abrir mesa" o "Cerrar mesa" según el estado) y Acciones (Editar / Borrar).

  • El formulario se controla con el estado mesaEditando (null = cerrado). Mismo patrón que productos.
  • Validación antes de mandar (front) y también en el back — requisito del TPI.
  • "Nuevo PIN" reusa el PUT con generarNuevoPin: true, sin abrir el formulario.
  • En la edición, el campo número está deshabilitado (no se cambia el número de una mesa existente).

Las llamadas al backend (table.services.js)

▼
src/components/admin/table.services.js 
// POST si es alta, PUT si es edicion. el numero va como entero en el alta.
export const guardarMesa = async (mesa) => {
  const isEdit = mesa.id;
  const url = isEdit ? `${API_URL}/api/mesas/${mesa.id}` : `${API_URL}/api/mesas`;
  const payload = { ...mesa };
  if (!isEdit) payload.numero = parseInt(payload.numero, 10);

  const res = await fetch(url, {
    method: isEdit ? 'PUT' : 'POST',
    headers: getHeaders(),
    body: JSON.stringify(payload)
  });
  if (!res.ok) {
    // uso el mensaje del backend (ej: mesa repetida) y si no hay, uno generico
    const data = await res.json().catch(() => ({}));
    throw new Error(data.mensaje || 'Error al guardar los datos de la mesa');
  }
  return res.json();
};

export const abrirMesa  = (id) => fetch(`${API_URL}/api/mesas/${id}/abrir`,  { method: 'POST', headers: getHeaders() });
export const cerrarMesa = (id) => fetch(`${API_URL}/api/mesas/${id}/cerrar`, { method: 'POST', headers: getHeaders() });

// regenera el pin reusando el PUT sin abrir el formulario
export const regenerarPin = (mesa) => guardarMesa({ ...mesa, generarNuevoPin: true });
Mejora que hicimos: antes guardarMesa mostraba siempre "Error al guardar..." aunque el back explicara el motivo. Ahora lee data.mensaje, así al duplicar una mesa aparece "Ya existe una mesa con el número X".

Validación (mismo criterio front y back)

src/components/admin/mesas.validations.js 
export const validarMesa = ({ numero }) => {
  const r = { error: false, mensaje: '' };
  if (numero === '' || isNaN(Number(numero)) || Number(numero) <= 0) {
    r.error = true; r.mensaje = 'El número de mesa debe ser un número mayor a 0';
  } else if (!Number.isInteger(Number(numero))) {
    r.error = true; r.mensaje = 'El número de mesa debe ser un número entero';
  }
  return r;
};

export const validarPin = (pin) => {
  const r = { error: false, mensaje: '' };
  if (!/^\d{4}$/.test(pin || '')) { r.error = true; r.mensaje = 'El PIN debe tener 4 dígitos'; }
  return r;
};

const handleValidar = async (e) => {

  } catch (err) { setError(err.message); }

El Context de mesa (estado global)

▼
src/services/mesa/MesaContextProvider.jsx 
export const MesaContextProvider = ({ children }) => {
  // arranco leyendo lo que haya en sessionStorage (si recarga, no pierde la mesa)
  const [mesaToken, setMesaToken] = useState(() => obtenerMesaToken());
  const [mesaId, setMesaId]       = useState(() => obtenerMesaId());
  const [numero, setNumero]       = useState(() => obtenerMesaNumero());

  const iniciarSesionMesa = (sesion) => {        // al validar el pin
    guardarSesionMesa(sesion);
    setMesaToken(sesion.mesaToken);
    setMesaId(String(sesion.mesaId));
    setNumero(String(sesion.numero));
  };

  const cerrarSesionMesa = () => {               // al cerrar o si el token es rechazado
    limpiarSesionMesa();
    setMesaToken(null); setMesaId(null); setNumero(null);
  };

  const sesionActiva = isTokenValid(mesaToken);  // hay sesion si el token existe y no expiro

  return (
    <MesaContext.Provider value={{ mesaId, numero, sesionActiva, iniciarSesionMesa, cerrarSesionMesa }}>
      {children}
    </MesaContext.Provider>
  );
};
src/services/mesa/mesa.session.js (persistencia en sessionStorage) 
// guarda token, id y numero de la mesa en sessionStorage
export const guardarSesionMesa = ({ mesaToken, mesaId, numero }) => {
  sessionStorage.setItem(MESA_TOKEN_KEY, mesaToken);
  sessionStorage.setItem(MESA_ID_KEY, String(mesaId));
  sessionStorage.setItem(MESA_NUMERO_KEY, String(numero));
};

Protección de rutas (importante para defender)

Dos niveles de protección distintos

▼
  • Rutas del personal (POST/PUT/DELETE /mesas): verificarToken + verificarRol(['super-admin','admin']).
  • Rutas del cliente (cuentas, carrito, pedidos): verificarMesaToken — valida el token de mesa y que la mesa del token sea la que se pide.
  • En el front, además, MesaProtected no deja entrar sin sesión válida.
backend/src/middlewares/auth.middleware.js 
// 1) token del personal: verifica el jwt del login y guarda el usuario en req
const verificarToken = (req, res, next) => {
  const token = (req.headers.authorization || '').split(' ')[1];
  try {
    req.usuario = jwt.verify(token, JWT_SECRET);
    next();
  } catch { return res.status(401).json({ mensaje: 'Token invalido o expirado' }); }
};

// 2) rol: recibe los roles permitidos y compara contra el del usuario
const verificarRol = (rolesPermitidos) => async (req, res, next) => {
  const rol = await Rol.findByPk(req.usuario.rolId);
  if (!rol || !rolesPermitidos.includes(rol.nombre)) {
    return res.status(403).json({ mensaje: 'Acceso denegado...' });
  }
  next();
};

// 3) token de mesa del cliente: que sea tipo mesa, la mesa ocupada y la sesion abierta
const verificarMesaToken = async (req, res, next) => {
  const token = (req.headers.authorization || '').split(' ')[1];
  try {
    const datos = jwt.verify(token, JWT_SECRET);
    if (datos.tipo !== 'mesa') return res.status(401).json({ mensaje: 'Token de mesa invalido' });

    const mesa = await Mesa.findByPk(datos.mesaId);
    if (!mesa || mesa.estado !== 'ocupada') return res.status(403).json({ mensaje: 'La mesa no esta habilitada' });

    const sesion = await SesionMesa.findOne({ where: { id: datos.sesionMesaId, mesaId: datos.mesaId, estado: 'abierta' } });
    if (!sesion) return res.status(403).json({ mensaje: 'La sesion de mesa no esta activa' });

    req.mesaCliente = { mesaId: datos.mesaId, numero: datos.numero, sesionMesaId: datos.sesionMesaId };
    next();
  } catch { return res.status(401).json({ mensaje: 'Token de mesa invalido o expirado' }); }
};

Posibles preguntas (con respuestas)

Las 15 preguntas más probables

▼
1. ¿Por qué un modelo SesionMesa aparte de Mesa?
Porque una mesa se usa muchas veces. Cada atención (de apertura a cierre) es una sesión distinta. Así los pedidos quedan ligados a la sesión (sesionMesaId), hay historial por atención, y "Mis pedidos" del cliente solo muestra los de la sesión actual.
2. ¿Cómo entra el cliente sin loguearse? ¿Diferencia con el token del personal?
El cliente no tiene usuario/contraseña: valida el PIN y recibe un token de mesa (JWT temporal, 8h). El token del personal sale del login con email/contraseña y lleva el rol. Son dos tokens distintos, validados por middlewares distintos (verificarMesaToken vs verificarToken).
3. ¿Por qué PIN si ya está el QR?
Porque cualquiera que conozca la URL (/numero) entraría sin estar en la mesa. El PIN funciona como segundo factor: el QR te lleva a la mesa, el PIN (que da el mozo y cambia en cada apertura) te autoriza. Al cerrar la mesa el PIN se invalida ('0000').
4. ¿Qué pasa con cuentas y pedidos al cerrar la mesa?
La sesión pasa a cerrada (con fechaCierre) y todas las cuentas abiertas de esa mesa se cierran. Los pedidos quedan guardados (asociados a esa sesión) como historial; no se borran.
5. ¿El QR se guarda en la base?
No. Se arma al vuelo con la URL de la mesa (origin/numero) usando un servicio externo (api.qrserver.com). Si cambia el dominio, se regenera solo.
6. ¿Qué pasa si dos personas escanean el mismo QR?
Las dos entran a la misma mesa (mismo PIN/sesión) y cada una crea su propia cuenta dentro de la mesa. Por eso la mesa tiene muchas cuentas (Mesa.hasMany(Cuenta)).
7. ¿Dónde se evita que se repita el número de mesa?
En dos lados: numero es unique en el modelo, y el controller chequea con un findOne antes de crear y devuelve un mensaje claro. Antes, sin ese chequeo, la constraint tiraba un 500 genérico.
8. ¿Qué estados puede tener una mesa y una sesión?
Mesa: disponible u ocupada (+ activa true/false). Sesión: abierta o cerrada.
9. ¿Por qué el token de mesa expira (8h)? ¿Y si expira mientras pide?
Para que el acceso no quede abierto para siempre. Si expira (o el back lo rechaza con 401/403), el front detecta sesionExpirada, cierra la sesión vía Context y manda de nuevo al PIN.
10. ¿Por qué Context? ¿Por qué sessionStorage y no localStorage?
Context porque es un estado global que usan varias pantallas (cumple el requisito del TPI). sessionStorage porque la sesión de mesa es temporal: se borra al cerrar la pestaña, así no queda "pegada" en el dispositivo del cliente (el login del personal sí usa localStorage).
11. ¿Qué pasa si el cliente recarga la página?
La sesión persiste en sessionStorage; el MesaContextProvider la vuelve a leer al arrancar (en los useState iniciales), así que no se pierde el acceso.
12. ¿Qué rol puede gestionar las mesas?
El alta/edición/borrado lo protege verificarRol(['super-admin','admin']). (Dato fino: abrir/cerrar mesa solo usan verificarToken, o sea cualquier usuario logueado — y como solo se loguean admin/super-admin, en la práctica son ellos.)
13. ¿El PIN de 4 dígitos es seguro?
Es simple y se puede adivinar (10.000 combinaciones). Mitiga que: cambia en cada apertura, se invalida al cerrar, y además haría falta la URL de esa mesa. Como mejora futura se podría limitar intentos o usar más dígitos.
14. ¿Hay validación de formularios?
Sí, doble: en el front (mesas.validations.js) antes de llamar, y en el back (validarMesa) antes de tocar la BD — mismo criterio, para avisar el error al usuario (requisito del TPI).
15. (La clásica) Explicá el flujo de punta a punta.
Admin crea la mesa → la abre (genera PIN + sesión) → el cliente escanea el QR y pone el PIN → el back valida y devuelve el token de mesa → el cliente arma su cuenta y pide → el admin ve los pedidos y al final cierra la mesa (cierra sesión y cuentas).
